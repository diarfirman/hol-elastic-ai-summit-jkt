Introduction & Architecture

Welcome to the Talk to Your Data hands-on lab. Today, we are breaking down AI silos by integrating Elastic Agent Builder with Google's Gemini Enterprise using the open A2A (Agent-to-Agent) protocol.

The Objective

You will build a specialized "eCommerce Data Analyst" Agent in Elastic that has direct access to your data. Instead of moving petabytes of data to an LLM, you will expose this Agent via the A2A protocol and connect it directly to the Gemini Enterprise UI.

Lab Syllabus & What You Will Learn

By completing this lab, you will learn how to bridge the gap between enterprise search and generative AI without moving your data.

  • Environment Provisioning: Spin up an Elastic Cloud cluster and ingest sample eCommerce datasets.
  • Deterministic AI Tools: Create Elastic Agent tools powered by ES|QL for accurate, real-time data aggregation.
  • Skill Engineering: Define behavioral guidelines (Skills) dictating how the agent formats and presents financial reports.
  • A2A Protocol Export: Assemble the Agent, Tools, and Skills into a standardized A2A (Agent-to-Agent) JSON payload.
  • Gemini Integration: Connect your specialized Elastic Agent to Google Gemini Enterprise to enable autonomous multi-agent analytics.
  • Agentic Actions: Trigger Kibana Workflows with Human-in-the-Loop directly from Gemini to execute marketing campaigns.

Part 2: Building the Elastic Agent

Follow this comprehensive 6-step roadmap to build, provision, and export your eCommerce Agent. You can choose to configure either via the interactive Kibana UI or using Dev Tools APIs.

STEP 1

Generate the API Key

The Agent and external platform (Gemini) need authorized access to read the eCommerce index and execute MCP commands. We will generate a least-privilege API key.

  1. Navigate to Stack Management > API keys from the left sidebar.
  2. Click on the blue Create API key button at the top right.
  3. Name the key: a2a-ecommerce-api-key.
  4. Select Restrict privileges and switch the editor input to JSON config. Paste the cluster permissions, index read controls, and space application feature locks (provided in Step 1 API query block below).
  5. Set the expiration parameter to 30 days.
  6. Click Create API key and copy the Base64 value returned.

Navigate to Management > Dev Tools in Kibana. Paste and run the following JSON payload:

POST /_security/api_key
{
  "name": "a2a-ecommerce-api-key",
  "expiration": "30d",
  "role_descriptors": {
    "mcp-access": {
      "cluster": ["monitor_inference"],
      "indices": [
        {
          "names": ["kibana_sample_data_ecommerce"],
          "privileges": ["read", "view_index_metadata"]
        }
      ],
      "applications": [
        {
          "application": "kibana-.kibana",
          "privileges": ["feature_agentBuilder.read", "feature_actions.read"],
          "resources": ["space:default"]
        }
      ]
    }
  }
}

Important: Copy and Save the Response

Save the encoded value from the JSON response immediately. This key will be used as the Bearer Token in Gemini Enterprise later.

Authorization Header Example:

The header format you will use to register the agent in the next module is:

STEP 2

Create a Custom ES|QL Tool

Next, we will create a custom tool powered by ES|QL (Elasticsearch Query Language). This tool calculates total revenue and order counts grouped by continent, using a dynamic ?startTime parameter that the LLM will provide.

  1. Navigate to Agent Builder > Tools from the left Kibana sidebar.
  2. Click on Create tool at the top right of the dashboard.
  3. Set the tool type to ES|QL query.
  4. Provide the tool ID: esql-revenue-by-region.
  5. Write the query logic inside the console: 
    FROM kibana_sample_data_ecommerce 
| WHERE order_date >= ?startTime 
| STATS total_revenue = SUM(taxful_total_price), total_orders = COUNT(*) BY geoip.continent_name 
| SORT total_revenue DESC 
| LIMIT ?limit
  6. Configure tool variables below the query console:
    • startTime: Date format, described as: Start time for the analysis in ISO format.
    • limit: Integer format, described as: Maximum number of results to return.
  7. Click Save to commit the tool to the library.

Run the tool registration query in your Dev Tools Console:

POST kbn:/api/agent_builder/tools
{
  "id": "esql-revenue-by-region",
  "type": "esql",
  "description": "An ES|QL tool to analyze eCommerce total revenue and order counts grouped by continent, filtered by a start time.",
  "tags": ["ecommerce", "sales", "esql"],
  "configuration": {
    "query": "FROM kibana_sample_data_ecommerce | WHERE order_date >= ?startTime | STATS total_revenue = SUM(taxful_total_price), total_orders = COUNT(*) BY geoip.continent_name | SORT total_revenue DESC | LIMIT ?limit",
    "params": {
      "startTime": {
        "type": "date",
        "description": "Start time for the analysis in ISO format (e.g., 2026-01-01T00:00:00Z)"
      },
      "limit": {
        "type": "integer",
        "description": "Maximum number of results to return"
      }
    }
  }
}

  2. Click on Create Skill.
  3. Enter Name: eCommerce Reporting Guidelines.
  4. Provide Description: Rules for presenting eCommerce financial and operational data.
  5. In the **Instruction Content** box, enter the guidelines (provided in Step 3 API query block below).
  6. Click Save.

Run the following query in Dev Tools:

POST kbn:/api/agent_builder/skills
{
  "id": "ecommerce-presentation-skill",
  "name": "eCommerce Reporting Guidelines",
  "description": "Rules for presenting eCommerce financial and operational data.",
  "content": "When answering questions about eCommerce sales:\n1. Always format 'total_revenue' in USD currency (e.g., $1,500.25).\n2. Mention the 'total_orders' to give context on volume.\n3. Present the regional breakdown in a clean bulleted list or markdown table.\n4. Provide a brief 1-2 sentence business insight summarizing the top-performing region."

Part 4: Test Your Work

It's time to see interoperability in action. We will ask Gemini natural language questions, and it will route them to Elastic to fetch the actual data.

Scenario 1: Natural Language Analytics

In the Gemini Enterprise chat interface, ask the following question:

"Ask the eCommerce Analyst Agent: What is the top-selling clothing category from last week's data?"

What happens under the hood?

  1. Gemini reads the prompt and determines the Elastic Agent is the expert.
  2. Gemini sends an A2A request to the Elastic Protocol Endpoint.
  3. The Elastic Agent executes a data search against the kibana_sample_data_ecommerce index.
  4. Elastic returns the structured data to Gemini.
  5. Gemini formats the response into human-readable text for the user.
Scenario 2: Multi-Agent Handoff

Without leaving the chat, follow up with a creative task based on the data just retrieved:

"Based on the top-selling items you just found, please draft a promotional marketing email to send to our customers."

Why this matters:

This demonstrates true Agentic interoperability. The data retrieved by the Elastic Agent (the domain expert) is seamlessly passed as context back to Gemini's core LLM (the creative expert) to generate marketing copy—all within a single workflow, without any manual copy-pasting or context switching.

Part 5: Actionable Workflows (HITL)

Take your Agent from a "Data Reader" to an "Action Taker". We will trigger a Human-in-the-Loop (HITL) workflow directly from Gemini to send our marketing campaign draft, pausing for human approval.

STEP 1

Create a Marketing Approval Workflow in Kibana

We will build a simple workflow that accepts email content as input, pauses to ask for your permission, and then simulates sending an email.

  • Navigate to Management > Workflows in Kibana.
  • Click Create Workflow.
  • Switch to the YAML tab in the editor.
  • Paste the following YAML definition:
version: "1"
name: Marketing Campaign Approval
description: A HITL workflow that pauses for approval before sending a marketing email.
enabled: true
triggers:
  - type: manual
    inputs:
      properties:
        email_draft:
          type: string
          description: The generated email draft to be reviewed
        audience_email:
          type: string
          description: The email address to send the campaign to
      required:
        - email_draft
        - audience_email
steps:
  - name: ask_approval
    type: waitForInput
    with:
      message: |-
        Please review the email draft below and approve or reject:
        
        {{ inputs.email_draft }}
        
        Target audience: {{ inputs.audience_email }}
      schema:
        properties:
          approved:
            type: boolean
            description: Approve or reject the campaign
        required:
          - approved
  - name: check_approval
    type: if
    condition: "steps.ask_approval.output.response.approved : true"
    steps:
      - name: send_campaign_email
        type: email
        connector-id: Elastic-Cloud-SMTP
        with:
          to:
            - "{{ inputs.audience_email }}"
          subject: Marketing Campaign
          message: "{{ inputs.email_draft }}"
    else:
      - name: log_rejection
        type: console
        with:
          message: "Campaign rejected. Email was NOT sent to {{ inputs.audience_email }}"
  • Click Save, and then make sure to click Enable.

Run the following query in Dev Tools to create the workflow programmatically:

POST kbn:/api/workflows
{
  "name": "Marketing Campaign Approval",
  "description": "A HITL workflow that pauses for approval before sending a marketing email.",
  "enabled": true,
  "workflow": "version: \"1\"\nname: Marketing Campaign Approval\ndescription: A HITL workflow that pauses for approval before sending a marketing email.\ntriggers:\n  - type: manual\ninputs:\n  email_draft:\n    type: string\n    description: The generated email draft to be reviewed\n  audience_email:\n    type: string\n    description: The email address to send the campaign to\nsteps:\n  - id: ask_approval\n    name: Wait for Approval\n    type: waitForInput\n    with:\n      message: \"Do you approve this marketing email draft to be sent to {{ inputs.audience_email }}?\"\n      schema:\n        type: object\n        properties:\n          approved:\n            type: boolean\n  - id: send_email\n    name: Send Email\n    type: email\n    connector-id: \"elastic-cloud-email\"\n    with:\n      to: [\"{{ inputs.audience_email }}\"]\n      subject: \"Exclusive Marketing Offer\"\n      message: \"{{ inputs.email_draft }}\"\n    if: \"${{ steps.ask_approval.output.approved == true }}\"\n  - id: reject_log\n    name: Log Rejection\n    type: console\n    with:\n      message: \"REJECTED! Email draft was not approved.\"\n    if: \"${{ steps.ask_approval.output.approved == false }}\""
}
STEP 2

Expose Workflow as a Tool

Before our agent can trigger this workflow, we must explicitly create a Tool for it and assign it to our Agent.

  • Navigate to Agent Builder > Tools and click Create Tool.
  • Name it Send Marketing Campaign and choose Workflow as the tool type.
  • Select the Marketing Campaign Approval workflow you just created. Click Save.
  • Now, navigate to Agent Builder > Agents and click your eCommerce Analyst agent.
  • Scroll down to the Tools section and click Add Tool.
  • Search for and select your new Send Marketing Campaign tool. (Note: Ensure the built-in tools platform.core.get_workflow_execution_status and platform.core.resume_workflow_execution are also attached so the agent can handle the HITL approval).
  • Click Save to update the agent.
STEP 3

Test the Workflow in Kibana Agent Builder

Before moving to Gemini, let's test if the agent can successfully orchestrate the workflow natively within Kibana.

  • While still in the eCommerce Analyst agent page, look at the Test your agent chat panel on the right.
  • Type: "Draft a promotional email for men's shoes and execute the Marketing Campaign Approval workflow to send it to my_email@example.com."
  • The agent will "think", call the execute_workflow tool, and then inform you that the workflow is waiting for approval.
  • Reply in the chat: "Yes, I approve the draft."
  • The agent will call resume_workflow_execution and confirm the email has been sent!
STEP 4

Trigger the Action from Gemini

Now that we've verified the Agent's internal logic, let's run the exact same process from our external Gemini Enterprise interface.

Scenario 3: Executing Agentic Actions Remotely

Send the following prompt in Gemini:

"Based on the men's shoe sales data rising, please create a promotional email draft. Then, execute the marketing workflow to send it to my_email@example.com, but wait for my approval first."

What happens under the hood?

  1. Gemini routes the complex request to the Elastic Agent via A2A.
  2. Agent generates the email draft using Gemini's reasoning.
  3. Agent uses execute_workflow to trigger your Kibana workflow, passing the draft as input.
  4. The Kibana workflow hits the "Wait for Input" step and pauses.
  5. Gemini informs you it is waiting for your approval. You can say "Yes, proceed" in Gemini.
  6. Agent translates your "Yes" into a resume_workflow_execution API call to Kibana to complete the workflow and send the email!
