Introduction & Architecture

Welcome to the Elastic Agent Builder + Gemini Enterprise hands-on lab. Today, we are breaking down AI silos by integrating Elastic Agent Builder with Google's Gemini Enterprise using the open A2A (Agent-to-Agent) protocol.

The Objective

You will build a specialized "eCommerce Data Analyst" Agent in Elastic that has direct access to your data. Instead of moving petabytes of data to an LLM, you will expose this Agent via the A2A protocol and connect it directly to the Gemini Enterprise UI.

Lab Syllabus & What You Will Learn

By completing this lab, you will learn how to bridge the gap between enterprise search and generative AI without moving your data.

  • Environment Provisioning: Spin up an Elastic Cloud cluster and ingest sample eCommerce datasets.
  • Deterministic AI Tools: Create Elastic Agent tools powered by ES|QL for accurate, real-time data aggregation.
  • Skill Engineering: Define behavioral guidelines (Skills) dictating how the agent formats and presents financial reports.
  • A2A Protocol Export: Assemble the Agent, Tools, and Skills into a standardized A2A (Agent-to-Agent) JSON payload.
  • Gemini Integration: Connect your specialized Elastic Agent to Google Gemini Enterprise to enable autonomous multi-agent analytics.
  • Agentic Actions: Trigger Kibana Workflows with Human-in-the-Loop directly from Gemini to execute marketing campaigns.
  • Agentic Actions: Trigger Kibana Workflows with Human-in-the-Loop directly from Gemini to execute marketing campaigns.

Architecture Overview

  • 1. Gemini Enterprise: Acts as the unified User Interface and Master Orchestrator.
  • 2. A2A Protocol: The open standard communication layer utilizing the Agent Card (JSON) and REST API endpoints.
  • 3. Elastic Agent Builder: The Domain Expert executing specialized tools (e.g., ES|QL queries) against your data.

Part 2: Building the Elastic Agent

Follow this comprehensive 6-step roadmap to build, provision, and export your eCommerce Agent. You can choose to configure either via the interactive Kibana UI or using Dev Tools APIs.

STEP 1

Generate an API Key

Agent and platform external (Gemini) need access to read the index and execute MCP.

Open Kibana Dev Tools and run:

POST /_security/api_key
{
  "name": "a2a-ecommerce-api-key",
  "expiration": "30d",
  "role_descriptors": {
    "mcp-access": {
      "cluster": ["monitor_inference"],
      "indices": [
        {
          "names": ["kibana_sample_data_ecommerce", "*"],
          "privileges": ["read", "view_index_metadata"]
        }
      ],
      "applications": [
        {
          "application": "kibana-.kibana",
          "privileges": ["feature_agentBuilder.read", "feature_actions.read"],
          "resources": ["space:default"]
        }
      ]
    }
  }
}
Note: Copy the encoded value from the response. This is your API Key! Kibana accept the format : headers "Authorization" = "ApiKey {encoded api key}"
STEP 2

Create a Custom ES|QL Tool

Let's build an ES|QL tool to query our eCommerce data.

  1. Navigate to Agent Builder > Tools > Manage All Tools > + New Tool or Type Agents / Tool in global search bar.
  2. Click New tool and select ES|QL query as the type.
  3. ID: esql-revenue-by-region.
  4. Paste the following query: 
    FROM kibana_sample_data_ecommerce 
| WHERE order_date >= ?startTime 
| STATS total_revenue = SUM(taxful_total_price), total_orders = COUNT(*) BY geoip.continent_name 
| SORT total_revenue DESC 
| LIMIT ?limit
  5. Configure variables: startTime (Date) and limit (Integer). Save. Or the easier way you can click Infer Parameters and change the data type accordingly.
  6. Click Save.

Run the following query in Dev Tools:

POST kbn:/api/agent_builder/tools
{
  "id": "esql-revenue-by-region",
  "type": "esql",
  "description": "An ES|QL tool to analyze eCommerce total revenue and order counts grouped by continent, filtered by a start time.",
  "tags": ["ecommerce", "sales", "esql"],
  "configuration": {
    "query": "FROM kibana_sample_data_ecommerce | WHERE order_date >= ?startTime | STATS total_revenue = SUM(taxful_total_price), total_orders = COUNT(*) BY geoip.continent_name | SORT total_revenue DESC | LIMIT ?limit",
    "params": {
      "startTime": {
        "type": "date",
        "description": "Start time for the analysis in ISO format (e.g., 2026-01-01T00:00:00Z)"
      },
      "limit": {
        "type": "integer",
        "description": "Maximum number of results to return"
      }
    }
  }
}
STEP 3

Create a Custom Agent Skill

Skills define behavioral guidelines for presenting financial reports.

  1. Go to Agent Builder > Skills > Manage All Skills. Or you can search in global search bar.
  2. Click + New Skill.
  3. ID: ecommerce-presentation-skill Name: eCommerce Reporting Guidelines.
  4. Add formatting instructions (see API payload below for the text) and Save.

Run the following query in Dev Tools:

POST kbn:/api/agent_builder/skills
{
  "id": "ecommerce-presentation-skill",
  "name": "eCommerce Reporting Guidelines",
  "description": "Rules for presenting eCommerce financial and operational data.",
  "content": "When answering questions about eCommerce sales:
1. Always format 'total_revenue' in USD currency (e.g., $1,500.25).
2. Mention the 'total_orders' to give context on volume.
3. Present the regional breakdown in a clean bulleted list or markdown table.
4. Provide a brief 1-2 sentence business insight summarizing the top-performing region."
}
STEP 4

Create the Agent & Assign Tools

Now we create the agent itself, give it instructions, and equip it with the tools we created.

  1. Go to Agent Builder > Agents and click Create Agent.
  2. Name: eCommerce Analyst.
  3. Provide system instructions defining its role.
  4. Add Tools: Check esql-revenue-by-region and all platform.core.* utilities. Save.

Run the following query in Dev Tools:

POST kbn:/api/agent_builder/agents

  "id": "elastic-ecommerce-agent",
  "name": "eCommerce Analyst",
  "description": "Hi! I am your eCommerce data analyst. Ask me about our sales revenue and regional performance.",
  "labels": ["ecommerce", "data-analyst", "esql"],
  "avatar_color": "#F04E98",
  "avatar_symbol": "EA",
  "configuration": {
    "instructions": "You are an expert eCommerce data analyst. You help users understand sales performance from the kibana_sample_data_ecommerce dataset. Use your ES|QL tools to fetch accurate data and apply your reporting skills to present it.",
    "tools": [
      {
        "tool_ids": [
          "esql-revenue-by-region",
          "platform.core.search",
          "platform.core.get_document_by_id",
          "platform.core.execute_esql",
          "platform.core.generate_esql",
          "platform.core.get_index_mapping",
          "platform.core.list_indices",
          "platform.core.generate_workflow",
          "platform.core.execute_workflow",
          "platform.core.list_workflow_executions",
          "platform.core.get_workflow_execution_status",

STEP 5

Assign the Skill via Kibana UI

Due to registry isolation between platform tools and analytical skills, we will visually bind the presentation guideline to our new agent inside the Kibana Portal.

Follow these clicks in the UI:

  • Navigate to the Agent Builder > Agents tab.
  • Select and open the newly created "eCommerce Analyst" agent card.
  • Scroll down to the Skills panel.
  • Click Add Skill.
  • Pick the "ecommerce-presentation-skill" skill created in Step 3.
  • Save the changes.
STEP 6

Verify & Export A2A Configuration

This final step is performed exclusively using Kibana Dev Tools. The A2A (Agent-to-Agent) protocol compiles our skills, tools, and schema definitions into a unified JSON format that external environments query directly.

Navigate back to Management > Dev Tools and run this GET query to fetch your fully assembled A2A Agent Card: 
GET kbn:/api/agent_builder/a2a/elastic-ecommerce-agent.json

Action Required:

Copy the entire JSON response payload from the right-hand panel of Dev Tools and save it locally on your machine. Name the file elastic-ecommerce-agent.json. This serves as your Agent Card.

STEP 7

Test the Agent in Kibana UI

Before integrating with Gemini, let's verify that the agent correctly understands your skills and executes the tools from within Kibana.

  • In the chat panel, ask:

  • You can see how the Agent is working and present the result

Part 3: Gemini Enterprise Integration

Gemini Enterprise natively authenticates using Google OAuth2, while Elastic A2A uses API Keys. We will deploy a lightweight Cloud Run proxy to seamlessly translate the authentication between them.

Reference: The deployment steps and architecture in this section are sourced from the open-source A2A Proxy project on GitHub.

1 Create a Google OAuth client

Gemini Enterprise needs a Google OAuth2 client ID/secret to authenticate end users.

  • Open an incognito window, go to console.cloud.google.com, and log in.
  • Search for credentials in the top bar and open APIs & Services → Credentials.
  • If prompted, create an OAuth consent screen (External or Internal).
  • Click + Create credentials and choose OAuth client ID.
  • Select Web application as the application type.
  • Name it Elasticsearch. Under Authorised redirect URIs, add: https://vertexaisearch.cloud.google.com/oauth-redirect. Click Create.
  • Copy the Client ID and Client Secret. You will need these for Gemini Enterprise later.

2 Deploy the A2A Proxy from Cloud Shell

  • Click the Cloud Shell terminal icon (>_) in the top right of the GCP console.
  • Wait for it to provision, then click Open editor in the Cloud Shell toolbar.
  • Open a terminal inside the editor and run: 
    git clone https://github.com/richardengineeringbrewery/datalabs-workshop-a2aproxy.git
cd datalabs-workshop-a2aproxy
  • Upload the elastic-ecommerce-agent.json file you exported from Kibana in Part 2. Rename it to agent-card.json and place it next to deploy.sh.
  • Open deploy.sh in the editor and replace the ELASTIC_API_KEY variable at the top with your encoded Elastic API Key.
  • Run the deployment script in the terminal: 
    ./deploy.sh
  • Wait for it to finish and copy the Service URL (e.g., https://a2aproxy-xxx.a.run.app).

3 Add the Agent to Gemini Enterprise

  • Open your Gemini Enterprise agent connection setup.
  • For the Agent Card URL, use your Cloud Run URL appended with the card path: <service-url>/.well-known/agent-card.json
  • Provide the Client ID and Client Secret from Step 1.
  • Construct your Authorization URL using your Client ID: 
    https://accounts.google.com/o/oauth2/v2/auth?client_id=YOUR_CLIENT_ID&redirect_uri=https://vertexaisearch.cloud.google.com/oauth-redirect&response_type=code&prompt=consent&access_type=offline
  • Save the connection and authorize it. You should now see eCommerce Analyst available to be invoked in Gemini.

Part 4: Test Your Work

It's time to see interoperability in action. We will ask Gemini natural language questions, and it will route them to Elastic to fetch the actual data.

Scenario 1: Natural Language Analytics

In the Gemini Enterprise chat interface, ask the following question:

"Ask the eCommerce Analyst Agent: What is the top-selling clothing category from last week's data?"

What happens under the hood?

  1. Gemini reads the prompt and determines the Elastic Agent is the expert.
  2. Gemini sends an A2A request with an OAuth2 token to the Cloud Run A2A Proxy.
  3. The A2A Proxy bypasses the OAuth token, injects the Elastic API Key, and forwards it to the Elastic Protocol Endpoint.
  4. The Elastic Agent executes a data search against the kibana_sample_data_ecommerce index.
  5. Elastic returns the structured data to Gemini via the proxy.
  6. Gemini formats the response into human-readable text for the user.
Scenario 2: Multi-Agent Handoff

Without leaving the chat, follow up with a creative task based on the data just retrieved:

"Based on the top-selling items you just found, please draft a promotional marketing email to send to our customers."

Why this matters:

This demonstrates true Agentic interoperability. The data retrieved by the Elastic Agent (the domain expert) is seamlessly passed as context back to Gemini's core LLM (the creative expert) to generate marketing copy—all within a single workflow, without any manual copy-pasting or context switching.

Part 5: Actionable Workflows (HITL)

Take your Agent from a "Data Reader" to an "Action Taker" by triggering a Human-in-the-Loop (HITL) workflow directly from Gemini.

STEP 1

Create a Marketing Approval Workflow in Kibana

  • Navigate to Workflows Tab and Click Create Workflow.
  • Switch to the YAML tab in the editor.
  • Paste the following YAML definition and click Save and Enable:
version: "1"
name: Marketing Campaign Approval
description: A HITL workflow that pauses for approval before sending a marketing email.
enabled: true
triggers:
  - type: manual
    inputs:
      properties:
        email_draft:
          type: string
          description: The generated email draft to be reviewed
        audience_email:
          type: string
          description: The email address to send the campaign to
      required:
        - email_draft
        - audience_email
steps:
  - name: ask_approval
    type: waitForInput
    with:
      message: |-
        Please review the email draft below and approve or reject:
        
        {{ inputs.email_draft }}
        
        Target audience: {{ inputs.audience_email }}
      schema:
        properties:
          approved:
            type: boolean
            description: Approve or reject the campaign
        required:
          - approved
  - name: check_approval
    type: if
    condition: "steps.ask_approval.output.response.approved : true"
    steps:
      - name: send_campaign_email
        type: email
        connector-id: Elastic-Cloud-SMTP
        with:
          to:
            - "{{ inputs.audience_email }}"
          subject: Marketing Campaign
          message: "{{ inputs.email_draft }}"
    else:
      - name: log_rejection
        type: console
        with:
          message: "Campaign rejected. Email was NOT sent to {{ inputs.audience_email }}"

STEP 2

Expose Workflow as a Tool

  • Navigate to Agents / Tools and click Create Tool.
  • Select Workflow as the tool type and choose Marketing Campaign Approval workflow.
  • Tool ID marketing-campaign, description 'workflows to create and send marketing-campaign', and click Save.
  • Go to Agent Builder > Agents and open your eCommerce Analyst agent.
  • In the Tools section, click Add Tool and select marketing-campaign. (Ensure platform.core.get_workflow_execution_status and platform.core.resume_workflow_execution are also attached).
  • Click Save.
STEP 3

Test the Workflow in Kibana Agent Builder

  • In the Test your agent chat panel, type:
    "Draft a promotional email for men's shoes and execute the Marketing Campaign Approval workflow to send it to my_email@example.com."
  • The agent will call execute_workflow and pause for approval.
  • Reply: "approve"
  • The agent will resume the workflow and send the email!
